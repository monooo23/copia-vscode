# Copia - Requirements

## 1. 背景

开发一个名为 `Copia` 的 VS Code 插件，针对当前激活的编辑器文件，在用户选中文本范围后，提供快速复制定位信息的能力。该产品的核心用途不是通用分享，而是将代码上下文快速整理后提供给 AI 助手。

主要使用场景包括：

- 向 AI 提问某段代码的含义、问题或修改方案
- 向 AI 提供精确文件位置，避免仅粘贴零散代码片段
- 在 AI 对话中补充多个相关文件或文件夹路径
- 将当前编辑上下文快速整理成适合 AI 消化的定位信息

核心目标是降低“手动整理文件路径 + 行号/列号范围并发送给 AI”的操作成本。

## 2. 产品目标

用户在当前激活文件中选中一段内容后，能够通过 VS Code 原生且可发现的入口快速触发复制操作，并一键复制适合发送给 AI 的定位信息到系统剪贴板。首选入口包括编辑器右键菜单、命令面板，以及在适用场景下的原生编辑器动作入口：

1. 仅文件路径
2. 文件路径 + 选择行范围
3. 文件路径 + 选择行范围 + 字符范围
4. 文件定位 + 选中代码内容

同时，用户在资源级入口中也能快速复制路径，用于向 AI 补充相关上下文：

1. 在资源管理器中右键文件或文件夹时复制路径
2. 在 Git Change / SCM 变更列表中右键文件时复制路径
3. 在资源管理器中多选文件、文件夹后批量复制路径

此外，当代码存在诊断项时，用户还应能快速复制适合发送给 AI 的诊断上下文：

1. 复制诊断消息
2. 复制出错代码位置
3. 复制“错误消息 + 文件路径 + 行列范围”的组合内容
4. 复制“错误消息 + 文件路径 + 行列范围 + 相关代码片段”的组合内容

## 3. 范围定义

### 3.1 MVP 范围

- 仅支持当前 active editor 中的文本文件
- 仅支持单个连续选区
- 选中非空范围后展示快捷操作
- 点击按钮后写入系统剪贴板
- 复制成功后给出明确提示
- 支持复制“文件定位 + 选中代码内容”的 AI 上下文格式
- 在资源管理器右键菜单中提供复制路径入口
- 在 SCM / Git Change 列表右键菜单中提供复制路径入口
- 支持多选文件批量复制路径
- 支持文件夹复制路径
- 支持文件与文件夹混合选择后批量复制路径
- 支持复制当前诊断项的消息与位置
- 支持基于 VS Code 诊断信息快速复制诊断上下文
- 支持复制诊断上下文时附带相关代码片段

### 3.2 非 MVP 范围

- 多选区 / 多光标合并复制
- 自定义复制模板
- 项目相对路径 / 绝对路径切换配置
- 支持 GitHub / Markdown / IDE deep link 等多种格式
- Webview 复杂浮层 UI

### 3.3 MVP+1 建议范围

以下能力与“快速复制上下文给 AI”高度相关，建议作为 MVP 后的优先增强：

- 在选区基础上自动附带前后若干行上下文
- 多选文件后，按统一格式打包多个文件的路径和代码内容
- 提供面向 AI 的预设复制模板，例如 `Explain`、`Debug`、`Refactor`、`Review`
- 对超长内容进行截断、预算控制和提示
- 对疑似敏感信息做可选脱敏处理

## 4. 用户故事

### 4.1 主用户故事

作为开发者，我希望在 VS Code 中选中某段代码后，能通过原生且顺手的入口立即复制文件定位信息，这样我可以直接把准确上下文粘贴给 AI，而不是手动整理文件路径和范围。

### 4.2 细化故事

- 作为开发者，我选中某段代码后，希望一键复制文件路径，以便告诉 AI 相关文件在哪里。
- 作为开发者，我选中多行代码后，希望一键复制文件路径和行范围，以便 AI 能快速定位问题区间。
- 作为开发者，我选中部分字符后，希望一键复制文件路径、行范围和字符范围，以便 AI 能理解更精确的上下文边界。
- 作为开发者，我选中一段代码后，希望一键复制“文件定位 + 代码内容”的完整上下文，直接粘贴给 AI。
- 作为开发者，我希望复制成功后能立即知道结果，避免重复点击。
- 作为开发者，我希望在资源管理器中右键文件或文件夹时，直接复制其路径，用来补充 AI 所需的关联上下文。
- 作为开发者，我希望在 Git Change 列表中右键变更文件时，也能直接复制路径，用来告诉 AI 当前改动涉及哪些文件。
- 作为开发者，我希望一次选择多个文件或文件夹，并批量复制它们的路径，便于给 AI 提供更完整的上下文范围。
- 作为开发者，我希望在代码报错时，能快速复制错误消息和出错位置，直接发给 AI 分析。
- 作为开发者，我希望在代码报错时，能通过 VS Code 原生的错误处理入口触发复制，而不是再手动整理报错内容。
- 作为开发者，我希望复制时能顺带包含实际代码片段，而不是只有路径和行号，这样 AI 更容易直接回答。
- 作为开发者，我希望插件能自动带上前后文几行代码，避免我每次都手动扩大选区。
- 作为开发者，我希望多选几个相关文件后，能一次性打包成适合发给 AI 的内容。

## 5. 功能需求

### 5.1 激活条件

插件应支持三类激活上下文：

- 编辑器选区上下文
- 资源右键上下文
- 诊断上下文

编辑器选区上下文应在以下条件同时满足时启用快捷操作：

- 当前存在 active text editor
- 当前文档为可读取文本文件
- 当前选区非空
- 当前选区属于单个连续范围

资源右键上下文应在以下条件下启用对应菜单项：

- 在 Explorer 中右键文件
- 在 Explorer 中右键文件夹
- 在 Explorer 中多选文件或文件夹
- 在 SCM / Git Change 列表中右键文件
- 在 SCM / Git Change 列表中多选文件

诊断上下文应在以下条件下启用对应菜单项或命令入口：

- 当前光标命中存在诊断项的位置
- 当前文件中存在可解析的诊断项，且用户通过命令面板主动触发
- 若扩展实现了 hover 增强，则 hover 命中存在诊断项的位置

### 5.2 编辑器入口要求

当用户选中范围后，插件应优先通过符合 VS Code 习惯的入口提供操作。MVP 推荐主入口如下：

- `editor/context` 中的复制命令
- `Command Palette` 中的复制命令

编辑器上下文至少包括以下能力：

1. `Copy Path`
2. `Copy Path + Lines`
3. `Copy Path + Lines + Chars`
4. `Copy Context`

说明：

- 当编辑器内命令数较多时，优先使用 `Copia` 子菜单，而不是平铺过多菜单项
- 命令面板中的名称应使用统一前缀，例如 `Copia: Copy Context`
- 若后续希望提供更接近选区的快捷入口，可作为可选增强采用：
  - CodeLens 样式的行内操作
  - Hover 中的命令入口
  - Editor decoration 附带命令
- 上述接近选区的入口不是 MVP 必选项
- 当前实现允许通过 CodeLens 作为补充入口直接触发选区复制能力

MVP 需优先保证稳定性、可发现性和与 VS Code 原生交互的一致性，不要求实现自定义浮动工具条。

### 5.3 资源右键菜单要求

插件应在以下位置展示 `Copy Path` 操作：

- Explorer 文件右键菜单
- Explorer 文件夹右键菜单
- SCM / Git Change 文件右键菜单

显示规则：

- 选中单个文件时显示 `Copy Path`
- 选中单个文件夹时显示 `Copy Path`
- 选中多个资源时仍显示 `Copy Path`
- 混合选择文件和文件夹时仍显示 `Copy Path`

行为规则：

- 单选时，复制该资源的路径
- 多选时，复制所有选中资源的路径
- 选择文件夹时，复制文件夹自身路径，不递归展开子文件
- 在 SCM / Git Change 列表中，复制的是资源文件路径，而不是 Git 状态文本
- 若未来资源级命令增加，应优先收敛到 `Copia` 子菜单，避免堆积过多右键操作

### 5.4 诊断上下文要求

插件应支持在存在诊断项的位置快速触发 `Copy Diagnostic` 和 `Copy Diagnostic + Code` 操作。

主入口应优先采用：

- `Code Action / Quick Fix` 样式入口
- 编辑器右键菜单中的命令入口
- 命令面板中的当前错误复制命令

可选增强入口包括：

- 自定义 hover 中的命令入口

约束说明：

- MVP 目标是“通过 VS Code 原生错误处理入口快速触发复制”
- 不要求将按钮直接插入 VS Code 原生诊断 hover 卡片内部
- 错误信息来源以 VS Code Diagnostics API 返回内容为准，不要求与终端 `tsc` 原始输出逐字一致
- 当前实现同时支持通过诊断行上的 CodeLens 入口直接复制诊断信息

触发规则：

- 当前光标所在位置若命中一个或多个诊断项，应允许复制诊断上下文
- 若扩展实现了 hover 增强，hover 位置命中诊断项时可展示辅助入口
- 若同一位置存在多个错误，MVP 可优先复制第一个诊断项，或弹出快速选择让用户选取

### 5.5 AI 上下文增强方向

虽然不属于 MVP，但产品设计应为以下 AI 场景保留扩展空间：

- `Copy With Surrounding Context`
  基于当前选区，自动附带前后若干行代码上下文
- `Copy Diagnostic + Code`
  在复制诊断消息与位置时，同时附带触发诊断的相关代码片段
- `Copy Multi-File Bundle`
  多选文件后，将多个文件内容按统一结构拼接，便于直接发送给 AI
- `AI-Friendly Templates`
  基于不同目的生成不同文案结构，例如解释、调试、重构、评审
- `Size Control`
  当复制内容过长时，自动截断、压缩或提示用户
- `Sensitive Data Redaction`
  对潜在密钥、token、邮箱、URL 参数等信息进行可选脱敏

这些能力在架构上应尽量复用已有的路径、范围、诊断和剪贴板模块。

### 5.6 复制内容格式

#### 5.6.1 单个资源复制文件路径

输出内容：

```text
path/to/file.ts
```

文件夹示例：

```text
path/to/folder
```

#### 5.6.2 多个资源复制路径

输出内容：

```text
src/a.ts
src/b.ts
docs/spec
```

规则：

- 每个资源占一行
- 输出顺序以 VS Code 提供的已选资源顺序为准
- 如果宿主 API 未提供稳定选择顺序，则以触发右键的资源为首项，其余资源按扩展拿到的顺序输出

#### 5.6.3 复制文件路径 + 选择行范围

输出内容：

```text
path/to/file.ts:10-18
```

规则：

- 行号使用 1-based
- 单行选择输出 `path/to/file.ts:10`
- 多行选择输出 `path/to/file.ts:10-18`

#### 5.6.4 复制文件路径 + 选择行 + 字符范围

输出内容：

```text
path/to/file.ts:10:5-18:22
```

规则：

- 行号使用 1-based
- 字符位置使用 1-based
- 格式定义为 `path:startLine:startChar-endLine:endChar`
- 当选择范围仅覆盖单行时，仍保持统一格式，例如：

```text
path/to/file.ts:10:5-10:22
```

#### 5.6.5 复制文件定位 + 选中代码内容

推荐输出内容：

```text
file: @/path/to/file.ts#L12:12-L31:123
```

```ts
const value = getUserName(user)
```

规则：

- 第一段固定为单行文件定位信息
- 格式定义为 `file: {pathRef}#L{startLine}:{startChar}-L{endLine}:{endChar}`
- 路径在工作区内时使用 `@/` 前缀表示工作区相对路径，例如 `@/src/app.ts`
- 路径不在工作区内时使用绝对路径，例如 `file: /tmp/foo.ts#L1:1-L3:10`
- 行号与字符位置均使用 1-based
- 第二段使用 fenced code block 包裹选中的原始代码内容
- 代码块语言优先根据文件扩展名推断，例如 `ts`、`tsx`、`js`、`py`
- 若无法推断语言，则使用无语言标记的 code fence
- 代码内容必须保持原始换行与缩进，不做格式化或裁剪

单行示例：

```text
file: @/src/app.ts#L24:7-L24:15
```

```ts
user.name
```

#### 5.6.6 复制诊断上下文

推荐输出内容：

```text
path/to/file.ts:10:5-10:22
error ts(2322): Type 'string' is not assignable to type 'number'.
```

规则：

- 第一行输出文件路径与诊断范围
- 第二行输出诊断消息
- 若诊断包含错误码和来源，则优先输出为 `severity source(code): message`
- 若无法稳定拿到错误码，则至少输出原始错误消息
- 行号与字符位置均使用 1-based

可接受示例：

```text
src/app.ts:24:7-24:15
error ts(2322): Type 'string' is not assignable to type 'number'.
```

```text
src/app.ts:24:7-24:15
warning eslint: Unexpected console statement.
```

#### 5.6.7 复制诊断上下文 + 代码片段

推荐输出内容：

```text
file: @/src/app.ts#L24:7-L24:15
error ts(2322): Type 'string' is not assignable to type 'number'.
```

```ts
user.name
```

规则：

- 第一行使用与 `Copy Context` 一致的 `file: ...#Lx:y-Lx:y` 定位格式
- 第二行为诊断消息
- 第三段起使用 fenced code block 包裹诊断范围对应的代码片段
- 若诊断范围为空，则至少复制当前诊断所在行文本
- 若代码片段超过配置的行数限制，则仅保留前若干行，并在代码块内追加折叠提示

### 5.7 路径规则

MVP 默认使用工作区相对路径：

- 若当前文件属于已打开工作区，则复制相对工作区根目录的路径
- 若当前文件不属于工作区，则回退为文件系统绝对路径
- 上述规则同样适用于 Explorer 和 SCM 中的文件、文件夹及批量复制
- 在 `Copy Context` 格式中，工作区相对路径使用 `@/` 前缀
- 在 `Copy Context` 格式中，工作区外文件直接使用绝对路径，不添加 `@/`

### 5.8 代码截断规则

当前实现支持对 `Copy Context` 与 `Copy Diagnostic + Code` 的代码块进行行数截断。

默认规则：

- 默认最多保留前 5 行代码
- 超出后，在代码块末尾追加一行折叠提示，例如 `... 85 more lines`
- 配置项为 `copia.maxCodeLines`
- 当配置为 `0` 时，表示禁用截断

### 5.9 剪贴板行为

点击任一操作按钮后，插件必须：

1. 生成目标字符串
2. 写入系统剪贴板
3. 给出成功提示

成功提示建议使用：

- 短暂状态栏提示
- 或低打扰的轻量提示

说明：

- 对高频复制操作，默认不应每次弹出 `InformationMessage`
- 仅在用户需要明确确认或发生异常时，再使用更显眼的提示方式

提示文案示例：

- `Copied file path`
- `Copied file path with line range`
- `Copied file path with line and char range`
- `Copied selection context`
- `Copied resource path`
- `Copied 3 paths`
- `Copied diagnostic context`
- `Copied diagnostic context with code`

### 5.10 失败处理

以下情况需要给出可理解的失败反馈：

- 无 active editor
- 选区为空
- 无可用资源选中
- 当前光标或 hover 位置无可复制诊断信息
- 剪贴板写入失败
- 当前文档 URI 不可解析为可复制路径
- 当前资源 URI 不可解析为可复制路径

失败提示应简洁明确，例如：

- `No active selection`
- `No resource selected`
- `No diagnostic found at current position`
- `Unable to copy to clipboard`
- `Current file path is unavailable`
- `Current resource path is unavailable`

## 6. 交互要求

### 6.1 响应时机

- 用户创建非空选区后，应尽快展示操作入口
- 用户取消选区或选区为空后，应隐藏操作入口
- 用户切换编辑器、切换文件或失去适用上下文后，应同步更新或隐藏入口
- 光标离开诊断位置或当前文件无诊断信息时，不显示诊断上下文复制入口

### 6.2 点击体验

- 点击后应立即执行，不再弹出二次确认
- 复制成功反馈应在 1 秒内可感知
- 重复选择不同范围时，展示内容应随最新选区更新
- 右键菜单触发复制时，不要求额外弹窗或输入
- 诊断上下文复制应优先保持在当前编辑流中完成，不打断用户操作

### 6.3 可发现性

首次安装后，用户无需额外配置即可使用核心能力，并能立即用于“复制上下文给 AI”这一核心场景。

所有首发入口应尽量贴合 VS Code 现有心智模型，避免引入风格突兀的自定义交互。

## 7. 边界与特殊情况

### 7.1 单行部分选中

如果用户只选中一行中的部分字符：

- `Copy Path` 仅复制路径
- `Copy Path + Lines` 只复制该单行行号
- `Copy Path + Lines + Chars` 复制精确到字符范围

### 7.2 整行多行选中

如果用户选中多行，行范围以选区起止行为准，不关心是否完整覆盖整行。

### 7.3 反向选择

当用户从后向前拖选时，输出应统一使用规范化后的起始位置和结束位置，不受拖选方向影响。

### 7.4 空选区

空选区不显示选区相关入口，不允许触发选区复制。

### 7.5 多工作区

若为 multi-root workspace：

- 默认取当前文件所属 workspace folder 的相对路径
- 不在任一 workspace folder 内时使用绝对路径

### 7.6 文件夹选择

选择文件夹时，仅复制该文件夹路径本身，不展开其内部文件列表。

### 7.7 混合多选

当同时选中文件和文件夹时：

- 允许一次性复制所有路径
- 每个选中资源单独占一行
- 不因资源类型不同而改变输出格式

### 7.8 SCM 资源选择

在 SCM / Git Change 列表中：

- 已修改、已暂存、未跟踪等可见文件资源都应尽可能支持复制路径
- 若某资源无法解析为文件系统路径，应跳过并给出失败提示，或整体失败并提示原因

### 7.9 多个诊断同时命中

当同一位置存在多个诊断项时：

- MVP 可默认复制首个诊断项
- 或通过 Quick Pick 让用户选择具体诊断项
- 结果必须可预测，避免随机选择不同错误

### 7.10 CodeLens 行内入口

当前实现中，CodeLens 可作为补充入口：

- 选区首行可显示 `Path`、`Lines`、`Chars`、`Context`
- 存在诊断的行可显示 `Copy Diagnostic`
- CodeLens 属于增强入口，主入口仍应以 context menu、command palette 和 code action 为主

### 7.11 诊断来源差异

诊断内容以 VS Code 当前可见诊断为准，可能来自：

- TypeScript / JavaScript 语言服务
- ESLint
- 其他语言服务器或诊断提供者

若用户期望的是 `tsc` 报错，MVP 应优先兼容 TypeScript 诊断格式，但不要求仅支持 `tsc`。

## 8. 非功能需求

### 8.1 性能

- 选区变化后的 UI 更新应足够轻量，避免明显卡顿
- 不应在每次光标移动时执行高成本操作

### 8.2 稳定性

- 不应因异常路径、只读文件、untitled 文档等场景导致插件报错或阻塞编辑器

### 8.3 兼容性

- 目标运行环境：桌面版 VS Code
- 兼容主流文本语言文件类型
- 输出格式应足够简洁稳定，便于粘贴到主流 AI 对话框中直接使用
- 诊断读取与 hover 扩展不应明显影响编辑器交互流畅度
- 后续增加代码片段、上下文拼接和多文件打包时，仍应保证可预测的输出结构
- CodeLens 刷新不应在大文件中造成明显性能抖动

### 8.4 可维护性

- 复制格式生成逻辑应独立封装，便于后续扩展更多模板
- 选区监听、UI 展示、剪贴板写入应职责分离

### 8.5 UX 一致性

- 首发交互应优先复用 VS Code 原生模式，例如 context menu、command palette、code action
- 避免在编辑器内引入与 VS Code 常见扩展风格明显不一致的重型浮层或工具条
- 命令名称、菜单分组和反馈方式应保持统一

## 9. 建议实现拆分

### 9.1 模块划分

- `selection tracker`: 监听当前编辑器和选区变化
- `range formatter`: 生成不同复制格式
- `context formatter`: 以适合 AI 消费的方式组织路径与范围信息
- `code block formatter`: 生成带语言标记的 code fence 输出
- `diagnostic resolver`: 读取当前位置诊断并提取消息、来源、错误码和范围
- `content collector`: 按需读取选区内容、周边上下文和多文件内容
- `template builder`: 生成适合 AI 消费的不同复制模板
- `redaction service`: 对敏感内容做可选脱敏
- `ui presenter`: 展示按钮或命令入口
- `clipboard service`: 写入剪贴板并反馈结果

### 9.2 推荐优先级

1. 实现路径与范围格式化逻辑
2. 实现剪贴板复制能力
3. 实现 `Copy Context` 的定位串与代码块格式化
4. 实现选区变化监听
5. 实现诊断读取与格式化
6. 接入 `editor/context` 与命令面板入口
7. 接入诊断上下文入口
8. 增加错误提示与边界处理
9. 设计并预留多文件 bundle 的扩展点

## 10. 验收标准

### 10.1 功能验收

- 在工作区内文件选择任意非空范围后，可通过 `editor/context`、命令面板或选区 CodeLens 触发 4 个复制操作
- 点击 `Copy Path` 后，剪贴板内容为相对路径
- 点击 `Copy Path + Lines` 后，剪贴板内容为正确的 1-based 行范围
- 点击 `Copy Path + Lines + Chars` 后，剪贴板内容为正确的 1-based 行列范围
- 点击 `Copy Context` 后，剪贴板内容包含正确的 `file: @/...#Lx:y-Lx:y` 定位串和原始选中代码
- 反向选择时，复制结果仍正确
- 清空选区后，操作入口消失
- 在 Explorer 中右键文件时，可看到 `Copy Path`
- 在 Explorer 中右键文件夹时，可看到 `Copy Path`
- 在 SCM / Git Change 列表中右键文件时，可看到 `Copy Path`
- 在 Explorer 中多选多个文件或文件夹后，可一次复制多条路径
- 混合选择文件和文件夹时，剪贴板内容按一行一个路径输出
- 选择文件夹时，复制的是文件夹路径本身
- 在存在诊断的位置，可触发 `Copy Diagnostic`
- 触发后，剪贴板包含诊断消息和正确的文件位置
- 当诊断带有 severity、code、source 时，复制结果优先包含这些信息
- 在存在诊断的位置，可触发 `Copy Diagnostic + Code`
- 触发后，剪贴板包含诊断消息、正确的文件位置以及相关代码片段
- 当代码片段超过配置限制时，输出结果应包含折叠提示
- 文档对后续 AI 增强能力有明确的结构预留，不会要求推翻现有格式设计

### 10.2 异常验收

- 没有选区时，不显示入口，也不会发生错误
- 非工作区文件能回退复制绝对路径
- 剪贴板写入异常时，用户能收到失败提示
- 没有资源选中时，不会发生错误
- SCM 中不可解析路径的资源不会导致扩展崩溃
- 当前光标位置没有诊断信息时，不会发生错误
- 同一位置多个诊断时，行为一致且可预测

## 12. 待确认项

当前按以下默认假设编写需求，如需调整应在实现前确认：

- 默认复制工作区相对路径，而不是绝对路径
- 行号和字符号均使用 1-based
- MVP 仅支持单个连续选区
- 接近选区或错误位置的快捷入口属于可选增强，不作为首发主入口
- 资源右键菜单阶段仅提供路径复制，不提供行范围和字符范围复制
- 诊断复制功能以 VS Code Diagnostics 为数据源，而不是终端 `tsc` 原始输出
- `Copy Context` 作为 MVP 正式能力，默认使用 `file: ...` + fenced code block 格式
- 首发版本优先采用 `editor/context`、`explorer/context`、`scm/resourceState/context`、`Code Action` 和 `Command Palette` 这类原生入口
- 当前产品名称为 `Copia`
- 当前实现通过 `copia.maxCodeLines` 控制代码块最大行数，默认值为 `5`
